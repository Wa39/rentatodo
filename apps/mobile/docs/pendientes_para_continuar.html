1. Dónde estamos

El esqueleto de la app ya existe y funciona: Expo + TypeScript en apps/mobile, 3 pestañas (Inicio · Mis rentas · Perfil), búsqueda por texto, detalle de artículo con calendario de disponibilidad y estados de reserva con sus colores. Todo el código ya usa la forma del contrato del PR #2 (snake_case, USD en centavos, enums en inglés, rangos de fechas), así que no habrá que rehacerlo cuando el contrato se apruebe. Los datos hoy son simulados detrás de una interfaz (FuenteDatos): al llegar el contrato congelado se escribe el cliente real y las pantallas no se tocan.

La rama feature/mobile-scaffold está lista en local, pendiente de mostrarse al equipo antes de subirla.

2. Lo que necesito de TRUCY (Backend · dueña del contrato)

2.1 🔴 Cerrar y congelar el contrato (PR #2)

Es LA dependencia: sin contrato congelado no se puede escribir el cliente real ni el flujo de reserva. Antes de aprobar, propongo resolver estas observaciones que salieron de revisar el borrador:

#ObservaciónPor qué importaPropuesta concreta
1El schema Error es solo un texto ({"error": "mensaje"}), sin códigos estables. El CANNOT_RENT_OWN_ITEM que ya habíamos acordado no aparece.Sin códigos, la app tendría que adivinar el error comparando textos: frágil y se rompe con cualquier cambio de redacción.Agregar campo code: {"error": {"code": "DATES_UNAVAILABLE", "message": "..."}}. Mínimo: CANNOT_RENT_OWN_ITEM, DATES_UNAVAILABLE, UNAUTHORIZED, TOKEN_EXPIRED, NOT_FOUND.
2LoginResponse trae solo access_token, sin refresh ni duración declarada.Cuando el token expire, la persona queda deslogueada a mitad de uso (p. ej. durante un check-in).O refresh token, o declarar duración larga para el curso (p. ej. 24 h) y documentarlo en el spec.
3POST /items/{id}/reservations sin idempotencia.Doble tap o reintento por mala señal = dos reservas idénticas.Header Idempotency-Key (UUID generado por la app), o que el backend rechace duplicado exacto (mismo renter+item+fechas) con 409.
4La descripción de unavailable_dates menciona estados en español ("solicitada, aprobada...") pero el enum es en inglés.Detalle de consistencia; confunde a quien lea el spec.Corregir la descripción a requested, approved, delivered, returned.

2.2 🔴 Tres respuestas cortas que el contrato aún no da

  1. ¿category acepta una sola o varias (?category=tools,camping)? — define si mis chips son de selección única o múltiple.
  2. ¿min_price/max_price son inclusivos y se puede mandar solo uno? — lo necesito para los chips < $10 y > $30.
  3. Para el polling: ¿cada cuánto es razonable consultar GET /reservations/{id} sin castigar al backend? (propongo cada 15 s con la pantalla abierta).

2.3 🟡 API corriendo

¿Cuándo habrá un API contra el cual apuntar la app, aunque sea local (docker compose up) o en un dev compartido? La app se configura con EXPO_PUBLIC_API_URL, solo necesito el valor.
