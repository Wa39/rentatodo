# User model + Auth — Design

## Context

`packages/contracts/openapi.yaml` and `apps/api` (FastAPI scaffold) are both merged
into `develop`. Item categories are decided. Per `apps/api/ROADMAP.md`, the next
piece of work is the first real domain model: `User`, plus the three Auth
endpoints (`POST /auth/register`, `POST /auth/login`, `GET /users/me`).

Mid-design, Jose supplied an external reference document (`CLAUDE_BACKEND.md`,
not committed to this repo) containing the team's full agreed conventions: all
6 database tables, the reservation state machine, the complete error-code
table, and the intended `apps/api` file layout. That document is authoritative
over ad-hoc decisions made earlier in this design session — where the two
conflicted, this design follows the document. Specific deviations from its
file tree are called out below, with the reasoning.

**Scope:** this design covers only `User` + the 3 Auth endpoints. Items,
Reservations, and everything past Week 1 are out of scope, even though
`CLAUDE_BACKEND.md` groups Auth+Items as a single "Week 1 (9 endpoints)" —
per `apps/api/CLAUDE.md`'s "don't implement beyond what's asked," Items is a
separate follow-up piece of work.

### Deviations from `CLAUDE_BACKEND.md`'s file tree (confirmed with Jose)

- **No `core/` module.** The document doesn't define one. Password hashing and
  JWT primitives (`hash_password`, `verify_password`, `create_access_token`,
  `decode_access_token`) live directly in `services/auth.py`, since the
  document describes `services/` as where business logic lives.
- **`app/exceptions.py` is new**, not in the document's tree. It's needed to
  translate errors into the document's mandated `{error: {code, message}}`
  shape (FastAPI's default is `{"detail": ...}`). This is cross-cutting
  infrastructure every future router will reuse, not an Auth-specific file.
- **`tests/routers/test_auth.py`**, not the document's flat `tests/test_auth.py`.
  Follows the existing convention already established by
  `tests/routers/test_health.py` and `apps/api/CLAUDE.md`'s testing section
  ("mirroring the structure of `app/`").
- **`dependencies/auth.py` ships only `get_current_user`** for this piece of
  work, not `get_optional_user` (also listed in the document's tree). None of
  the 3 Auth endpoints need optional auth; it's deferred until a future
  endpoint (e.g. `GET /items/{id}`) actually needs it, per "don't implement
  beyond what's asked."

## Known discrepancy for later (not blocking this work)

`CLAUDE_BACKEND.md`'s error-code table lists `403 CANNOT_RENT_OWN_ITEM` for
renting your own item, but the merged `openapi.yaml` documents that same case
as a generic `422` on `POST /items/{item_id}/reservations`. Flagged for
whoever picks up Reservations — not touched by this design.

## Data model — `User` (`app/models/user.py`)

| Column | Type | Notes |
|---|---|---|
| `id` | `UUID`, PK | Generated by Postgres (`server_default=gen_random_uuid()`) — native in PG13+, no extension needed on PG16 |
| `name` | `String`, not null | |
| `email` | `String`, not null, **unique** | |
| `password_hash` | `String`, not null | bcrypt hash. Never serialized in any response schema |
| `created_at` | `DateTime(timezone=True)`, `server_default=func.now()` | |

No `updated_at` — not in the contract's `UserResponse`, not needed yet (YAGNI).

## Configuration (`app/config.py`)

```python
class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_file=".env", extra="ignore")

    database_url: str
    jwt_secret: str
    jwt_expiration_hours: int = 24


settings = Settings()
```

`app/database.py` drops its own `Settings` class and imports `settings` from
`app.config` instead. `.env.example` gains:

```env
JWT_SECRET=change-me-in-production-use-random-64-chars
JWT_EXPIRATION_HOURS=24
```

## Password hashing + JWT (`app/services/auth.py`)

- `hash_password(plain: str) -> str` — `bcrypt.hashpw`, decoded to `str` for
  storage. bcrypt silently ignores any bytes past its 72-byte limit, so
  `RegisterRequest.password` gets an explicit `max_length=72` in the Pydantic
  schema — an over-long password is rejected with a normal `422` at the
  validation layer, instead of being silently truncated deeper in the stack.
- `verify_password(plain: str, hashed: str) -> bool` — `bcrypt.checkpw`.
- `create_access_token(user_id: UUID) -> str` — PyJWT, HS256, claims `sub`
  (user id), `iat`, `exp` (`now + jwt_expiration_hours`), signed with
  `settings.jwt_secret`.
- `decode_access_token(token: str) -> UUID` — decodes and validates,
  re-raising `jwt.ExpiredSignatureError` and `jwt.InvalidTokenError`
  separately so the caller (the dependency) can map them to different error
  codes (`TOKEN_EXPIRED` vs `UNAUTHORIZED`).

`requirements.txt` gains `bcrypt>=4.0,<5.0` and `PyJWT>=2.8,<3.0`.

## Error handling infrastructure (`app/exceptions.py`)

```python
class AppError(Exception):
    """Domain error mapping directly to the contract's error shape."""
    def __init__(self, status_code: int, code: str, message: str) -> None:
        self.status_code = status_code
        self.code = code
        self.message = message
```

A global handler registered on the FastAPI app (`main.py`) catches `AppError`
and returns `JSONResponse({"error": {"code": ..., "message": ...}}, status_code=...)`,
matching `CLAUDE_BACKEND.md`'s mandated error shape (same structure as the
`Error` schema already in the merged contract).

## `get_current_user` (`app/dependencies/auth.py`)

Uses `HTTPBearer(auto_error=False)` (not the default) so a missing header can
be mapped to our error shape instead of FastAPI's built-in one.

| Condition | Result |
|---|---|
| No `Authorization` header, or malformed | `401 UNAUTHORIZED` |
| Token signature invalid / corrupt | `401 UNAUTHORIZED` |
| Token well-formed but expired | `401 TOKEN_EXPIRED` |
| Token valid but user no longer exists | `401 UNAUTHORIZED` |
| Token valid, user exists | returns the `User` |

## Business logic (`app/services/auth.py`)

- `register_user(db, data: RegisterRequest) -> User` — pre-checks the email
  isn't taken, hashes the password, inserts. Also catches the unique-constraint
  `IntegrityError` as a defense against a race between two simultaneous
  registrations with the same email. Either path raises `422 VALIDATION_ERROR`
  (the document has no more specific code for this case).
- `authenticate_user(db, data: LoginRequest) -> tuple[User, str]` — looks up
  by email, verifies the password. Wrong email or wrong password both raise
  the same `401 UNAUTHORIZED` with a generic message — deliberately not
  revealing which one was wrong. Returns the user and a freshly created
  access token.

## Endpoints (`app/routers/auth.py`)

Thin — each maps directly to a service call and a schema:

- `POST /auth/register` → `201`, `UserResponse`
- `POST /auth/login` → `200`, `LoginResponse` (`access_token`, `token_type`,
  `expires_in` — all matching the merged contract)
- `GET /users/me` → `200`, `UserResponse`, requires `get_current_user`

## Schemas (`app/schemas/auth.py`)

`RegisterRequest`, `LoginRequest`, `LoginResponse`, `UserResponse` — mirror
`openapi.yaml` exactly, with `Field(..., description=...)` on non-obvious
fields per `apps/api/CLAUDE.md`.

## Testing (`tests/conftest.py`, `tests/routers/test_auth.py`)

New fixtures in `conftest.py` (project has no `conftest.py` yet):

- `db_session` — wraps each test in a transaction/savepoint against the real
  docker-compose Postgres, rolled back on teardown. No manual cleanup, no
  cross-test pollution.
- `client` — `TestClient(app)` with `get_db` overridden to use `db_session`,
  so requests inside a test run in the same isolated transaction.
- `make_user` — factory fixture (`make_user(email=..., password=...) -> User`)
  for tests that need an already-persisted user.

Tests, happy path + primary failure per `apps/api/CLAUDE.md`'s testing rule:

- **register:** 201 + created user has no `password_hash` in the response /
  422 on duplicate email
- **login:** 200 with token + `expires_in` / 401 on wrong password
- **me:** 200 with a valid token / 401 with no token / 401 `TOKEN_EXPIRED`
  specifically, using a JWT crafted with a past `exp`

## Migration

One new Alembic revision creating `users`: `id UUID PK DEFAULT gen_random_uuid()`,
`email` with a unique index, `password_hash`, `name`, `created_at TIMESTAMPTZ
DEFAULT now()`.
